Create a real-time web application for a **quiz buzzer competition** where multiple players join a game and compete to press a buzzer the fastest. The system must support a **Host** and multiple **Participants**, and everyone must be able to connect to the same session using a **Room Code**.

General Requirements

* The application must work in real time using WebSockets (for example with Socket.IO).
* The **server must determine the fastest buzzer press**, not the client.
* The interface must work smoothly on both **mobile phones and desktop browsers**.
* All state updates (buzz events, timers, resets) must instantly update for everyone connected.

Entry Screen
When a user opens the website, show a simple start screen with:

* A field to enter their **Name**.
* A field to enter a **Room Code**.
* Two buttons:

  * **Create Game (Host)**
  * **Join Game (Participant)**

Room System

* When the Host creates a game, the system generates a **unique Room Code** (for example: K8F21).
* The Host shares this code with other players.
* Participants enter the code to join the same game session.
* All players connected with the same Room Code must see the same game state in real time.

Host Interface
When someone joins as **Host**, show a dashboard with the following layout.

Teams Section

* There are **two teams**: Team A and Team B.
* Each team has **6 player slots**.
* Each slot displays the **name of the participant** assigned to that slot.
* When someone buzzes first, the Host must clearly see **which player pressed the buzzer first**.

Main Control Panel (center of screen)

Include three main buttons:

1. Reset Buzzers

   * Clears the current round.
   * Unlocks all buzzers so players can press again.

2. Correct Answer

   * Used when the buzzing player answers correctly.
   * Immediately resets the round and re-enables buzzers.

3. Wrong Answer

   * Used when the buzzing player answers incorrectly.
   * Triggers the next stage of the game logic.

Timer Display
The Host must also see a **countdown timer** that appears when a player buzzes.

Participant Interface

When a user joins as a **Participant**, show the following screen.

Team Selection

* Display **Team A and Team B**.
* Each team has 6 visible player slots.
* The participant selects a slot to join that team.
* The participant can switch teams easily by tapping another slot.

Buzzer Screen

* Display a **large buzzer button in the center of the screen**.
* The button must be very easy to press on mobile devices.
* When someone buzzes first, the buzzer must immediately lock for all other players.

Game Logic

Step 1 — First Buzz

* When a participant presses the buzzer:

  * The server determines the fastest player.
  * All other buzzers lock immediately.
  * Everyone sees the **name of the player who buzzed first**.
  * A **5-second countdown timer** appears.

Step 2 — Host Decision

If the player answers correctly:

* The Host presses **Correct Answer**.
* The system resets the buzzers.
* A new round begins.

If the player answers incorrectly:

* The Host presses **Wrong Answer**.
* The **opposite team gets a 10-second chance** to buzz.

Step 3 — Opponent Chance

During this 10-second window:

* Only players from the **other team** are allowed to press the buzzer.
* If someone buzzes:

  * A **5-second answer timer** appears.

If they answer correctly:

* The Host presses **Correct Answer**.
* The buzzers reset.

If they answer incorrectly:

* The Host presses **Wrong Answer**.

Step 4 — Open Buzz Phase

After both teams fail:

* The buzzers reopen for **all participants**.
* Anyone can buzz again.

If someone buzzes:

* A **5-second answer timer** appears.

If the answer is correct:

* Reset the buzzers and start a new round.

If the answer is wrong:

* Reset the buzzers immediately and start a new round.

Additional Features

* Show clearly which player buzzed first.
* Display countdown timers visually for everyone.
* Lock and unlock buzzers automatically based on game state.
* Show which team the buzzing player belongs to.
* Prevent multiple buzzes in the same round.
* Ensure all players receive updates instantly.

Technical Expectations

* Use a server (for example Node.js) to manage game state and buzzer logic.
* Use WebSockets (such as Socket.IO) for real-time communication.
* Do not rely on client timing for detecting the fastest buzzer.
* Maintain clear game states such as:

  * waiting for buzz
  * first buzz
  * opponent chance
  * open buzz
  * reset
